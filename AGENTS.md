# AGENTS.md

This file provides guidance to coding agents when working with code in this repository.

## Commands

```sh
npm run build        # tsc to dist/ + mark the CLI entrypoint executable (via node, so it works cross-platform)
npm run build:skill  # Regenerate skills/flutter-axi/SKILL.md from shared CLI guidance and SDK built-ins
npm run dev          # Run the CLI from source with tsx
npm test             # vitest run --dir test (unit suite, no devices needed)
npm run test:watch   # vitest watch mode
npm run test:e2e     # live suite - needs a booted simulator (two for multi-app tests); runs files serially
```

Run a single test file: `npm test -- test/cli.test.ts`.
Filter by test name: `npm test -- -t "parseFinderString"`.

The committed `skills/flutter-axi/SKILL.md` is generated by `npm run build:skill`; `npx tsx scripts/build-skill.ts --check` fails if it drifts.
Update the generator (`src/skill.ts`) instead of hand-editing `SKILL.md`.
`package.json` includes `skills/flutter-axi/` in `files`, so npm releases ship the generated installable Agent Skill.

The benchmark harness is a separate package under `bench/` (own `package.json`): `cd bench && npm run bench -- <run|matrix|report>`, `npx vitest run test` for its unit tests.

## Project Conventions

- Node 20+, TypeScript, ESM-only (`"type": "module"`, module resolution `Node16` - relative imports use `.js` extensions even from `.ts` files).
- Unit tests live in `test/*.test.ts`; live e2e tests in `e2e/*.e2e.test.ts` (gated: they skip when no simulator is booted). Both run with Vitest.
- Run `npm run build` and `npm test` before pushing.
- Do not hand-edit generated files: `skills/flutter-axi/SKILL.md` (owned by `build:skill`) and `bench/fixtures/counter_app/lib/flutter_axi_main.dart` (owned by `setup driver`).
- Captured Dart MCP output formats live in `test/fixtures/mcp-outputs.md` - parsers are written against those exact shapes; when the Dart SDK updates change a format, re-capture and update the fixture and its tests together.

## Architecture

flutter-axi is an agent-ergonomic CLI for driving Flutter apps, architecturally a sibling of [chrome-devtools-axi](https://github.com/kunchenguid/chrome-devtools-axi): it wraps the [Dart MCP server](https://dart.dev/tools/mcp-server) the way chrome-devtools-axi wraps chrome-devtools-mcp, plus a native device layer the browser tool has no equivalent of.
Every invocation is a short-lived process, so anything that must survive across commands lives in a detached bridge process or under `~/.flutter-axi/`.

### Process model

Three processes: CLI -> bridge -> `dart mcp-server` (which drives the running Flutter app via DTD / flutter_driver), plus direct `adb`/`xcrun simctl` child processes for the native layer (no bridge involved).

The CLI (`bin/flutter-axi.ts` -> `src/cli.ts`) parses args, calls MCP tools through the bridge, and formats output.
`ensureBridge` (`src/client.ts`) reads its session's `bridge.pid` and reuses a live bridge only after a **deep** health check (`/health?deep=1` drives one `list_running_apps` call); otherwise it spawns the bridge (`bin/flutter-axi-bridge.ts` -> `src/bridge.ts`) **detached** as a process group leader and polls health until the `FLUTTER_AXI_BRIDGE_TIMEOUT_MS` deadline (default 30s).
Every client entry point takes an optional explicit session parameter (used by the multi-app `run` command); when omitted the session resolves from `FLUTTER_AXI_SESSION`.

The bridge holds one persistent MCP stdio session and exposes a localhost HTTP API on its session port (9424 by default): `POST /call`, `GET /tools`, `GET /health[?deep=1]`.
`/call` prefers a tool's `structuredContent` (JSON) over its text blocks (`extractResultText`) - the Dart MCP server returns both and the parsers consume the JSON form - and forwards image content blocks (driver screenshots) as base64 in an `images` array.
Teardown kills the bridge's process group so `dart mcp-server` and the `flutter run` processes it spawned get reaped together.

**One bridge session = one Dart MCP child = one target app.** The Dart MCP server has a single "active application" notion, so multi-app orchestration maps to multi-session (`--app user`, `--app operator`), exactly the chrome-devtools-axi session model (`src/sessions.ts`: FNV-1a port derivation, validated names, per-name state dirs under `~/.flutter-axi/sessions/<name>/`).

### App lifecycle and the driver shim

`launch` (`src/lifecycle.ts`) calls `add_roots` (required - the Dart MCP server only operates under registered roots and discovers the Flutter SDK through them), then `launch_app`, parses `{dtdUri, pid}`, calls `connect_dart_tooling_daemon`, detects the app id from platform files, and persists everything to the session's `app.json` (`src/appstate.ts`) for later short-lived invocations (logs, stopapp, native commands).

**flutter_driver only works for apps launched through a driver-enabled entrypoint.** `setup driver <root>` idempotently adds `flutter_driver` to dev_dependencies and writes `lib/flutter_axi_main.dart` (calls `enableFlutterDriverExtension()` before the app's `main()`); `launch` auto-targets the shim when present. `attach --dtd <uri>` is inspection-only.

### Snapshots: the uid -> finder registry

The Dart MCP driver addresses widgets with *finders* (ByValueKey/ByText/ByType/...), not uids, so flutter-axi mints its own:
`takeSnapshot` (`src/actions.ts`) fetches `get_widget_tree` (JSON inspector nodes), `src/widgettree.ts` parses it and derives the best *simple* finder per node (ValueKey from the description annotation > unique text > unique non-structural type; ambiguous nodes render without a uid), `src/refs.ts` persists the uid->finder map to the session's `refs.json`, and the generation counter (`src/generation.ts`) stamps refs as `uid=g<N>:12`.
Action commands resolve refs through `resolveFinderArg`, which fails loudly with `STALE_REF` on generation mismatch and `REF_NOT_FOUND` on unknown uids; `text:`/`key:`/`type:`/`tooltip:`/`label:` finder strings bypass the registry entirely.
Nested Descendant/Ancestor finders are deliberately not derived - the Dart MCP server (SDK 3.11.5) forwards nested finder maps in Dart `toString()` form, which the driver's jsonDecode rejects (see `test/fixtures/mcp-outputs.md` for this and every captured output format).

Post-action snapshots wait `POST_ACTION_SETTLE_MS` (driver commands return when dispatched, not when the rebuild has painted), and `launch`/`attach` retry the first snapshot for up to 20s (`takeSnapshotWithRetry`) because the tree isn't servable until the first frame renders.

### Native layer

`src/device.ts` is pure command *builders* (exact argv, unit-testable without any toolchain) plus thin executors with an injectable `exec`.
`resolveAdb` probes ANDROID_HOME/ANDROID_SDK_ROOT/the default macOS SDK path/PATH; a missing toolchain surfaces as a structured `TOOLCHAIN_MISSING` error, never a raw ENOENT.
Friendly permission names map per platform (`PERMISSION_MAP`); iOS push writes a real `.apns` payload for `simctl push`, Android push is a local-notification approximation (documented limitation).

### The `run` script runner (multi-app)

`flutter-axi run` (`src/run.ts`) executes a stdin script with `app` (the invoking session) and `apps.<name>` (a Proxy lazily binding helpers to named sessions) globals - one script can orchestrate two apps on two devices.
Only the script's own `console.log` reaches stdout via the `RAW_STDOUT_MARKER`/`wrapStdout` mechanism in `src/cli.ts`.

### CLI output and AXI integration

Built on `axi-sdk-js` (`runAxiCli`); `--app <name>` is extracted from anywhere in argv (`extractAppFlag`) and exported as `FLUTTER_AXI_SESSION` before dispatch, so downstream modules need zero plumbing.
Output per command: TOON-encoded metadata block, indented widget tree with stamped uids, then a `help[N]:` block of contextual suggestions (`src/suggestions.ts` - suggested refs carry the stamped `g<N>:` prefix so agents copy back the exact printed form).
Errors are `FlutterAxiError` (extends `AxiError`) with codes `BRIDGE_NOT_READY | NO_APP | REF_NOT_FOUND | STALE_REF | TIMEOUT | DRIVER_ERROR | TOOLCHAIN_MISSING | VALIDATION_ERROR | UNKNOWN`.

## Things to know when editing

- The bridge resolves its own script path at runtime (`resolveBridgeScript`): it prefers a sibling `.ts` (dev mode, run via tsx) and falls back to the built `.js`.
- `resolveOutputPath` (`src/paths.ts`) is the chokepoint for caller-supplied output paths (screenshots, push payloads, GPS routes).
- `callToolIfRunning` deliberately never starts the bridge - the home view and SessionStart hook must stay cheap and side-effect free.
- Two `flutter run`s of the same project root on the same device contend (build dir + bundle id); stop other sessions' apps first. The e2e suite runs files serially (`--no-file-parallelism`) for this reason.
- e2e suites are gated on `bootedSimulators()` and the counter fixture; they must stay skippable so `npm test`/CI never needs a device.
