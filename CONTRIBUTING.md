# Contributing

flutter-axi follows the [AXI](https://github.com/kunchenguid/axi) design principles. Contributions should preserve those principles - token-efficient TOON output, minimal default schemas, structured errors with recovery suggestions, no interactive prompts, contextual next-step hints.

**Human-authored pull requests targeting `main` must be raised through [`no-mistakes`](https://github.com/kunchenguid/no-mistakes).**
A GitHub Actions check (`Require no-mistakes`) fails PRs whose body is missing the signature no-mistakes writes; the release and dependency bots are exempt.

## Workflow

1. Create a branch and make your changes.
2. Initialize the gate once (`no-mistakes init`), then push through it instead of pushing to `origin`:

   ```sh
   git push no-mistakes
   ```

   The pipeline reviews, tests, and builds in an isolated worktree, then opens the PR for you.
3. Run the local verification pass before pushing:

   ```sh
   npm run build
   npm test                                  # unit suite - no devices needed
   npx tsx scripts/build-skill.ts --check    # generated skill is current
   (cd bench && npx vitest run test)         # bench harness unit tests
   ```

3. For changes touching the bridge, snapshots, interactions, or the native layer, also run the live suite against a booted simulator:

   ```sh
   npm run test:e2e                          # two booted simulators enable the multi-app tests
   ```

4. Open a PR with a conventional-commit-style title (`feat:`, `fix:`, `docs:`, `refactor:`, `test:`).

## Repo conventions

- Node 20+, TypeScript, ESM-only (`.js` extensions in relative imports).
- Unit tests in `test/*.test.ts` (Vitest); live tests in `e2e/*.e2e.test.ts`, gated so they skip without a booted device.
- Do not hand-edit `skills/flutter-axi/SKILL.md` - it is generated by `npm run build:skill` from `src/skill.ts`; update the generator instead.
- Keep `skills/flutter-axi/` in the npm package `files` list so releases include the installable Agent Skill.
- Parsers for Dart MCP output are written against the captured formats in `test/fixtures/mcp-outputs.md`. If a Dart SDK update changes an output shape, re-capture it live, update the fixture, and adjust the parser and its tests in the same change.
- New commands must follow the AXI checklist: per-command `--help` entry in `COMMAND_HELP`, TOON output, structured `FlutterAxiError` with suggestions on every failure path, idempotent mutations (repeat = no-op success), `--app` compatibility, and a `test/` case for argument parsing.
- Native-layer additions go through pure command builders in `src/device.ts` (exact-argv unit tests for both platforms) with executors that use the injectable `exec`.

## Benchmark changes

The harness in `bench/` replicates [axi's bench-browser](https://github.com/kunchenguid/axi) methodology. When adding tasks or conditions:

- Tasks that only one condition can perform must declare `applicable_conditions` so other conditions are reported N/A, never failed.
- Update `bench/README.md`'s fairness caveats when a change affects comparability.
- Regenerate published results only from a full `matrix` run; per-run artifacts (agent output, judge output, grades) land under `bench/results/` for auditability.
